Ursa for Kafka 与 Diskless/Lakehouse Streaming 竞品深度比较

日期：2026-05-28

调研问题

在前一份 Diskless Kafka、Lakehouse/Iceberg 与学术前沿调研 的基础上，进一步深读 StreamNative Ursa / Ursa-for-Kafka，并和合适的竞品做深度比较：它到底解决什么问题，优势、缺点、trade-off 是什么，应该和哪些产品或架构路线放在同一张图里比较。

方法与来源

按工作区 deep research survey 流程，先做 Tavily broad source discovery，再用一手资料和既有报告交叉校验。初始 Tavily 查询包括：

  • Ursa for Kafka StreamNative lakehouse native diskless Kafka competitors WarpStream AutoMQ Bufstream Aiven Inkless
  • diskless Kafka KIP-1150 WarpStream AutoMQ Bufstream Ursa Iceberg Tableflow comparison
  • Kafka topic as Iceberg table Bufstream Ursa Confluent Tableflow Aiven Iceberg Topics

主要来源类型：

  • Ursa 一手资料：PVLDB 2025 论文、StreamNative UFK/Lakestream 博客和文档。
  • Kafka 上游资料：KIP-1150、KIP-1163、KIP-1164、KIP-1183，以及 Aiven 对 KIP-1150 accepted 的说明。
  • 竞品资料：WarpStream docs/blog/Tableflow、AutoMQ docs/Table Topic、Confluent Freight/Tableflow docs/blog、Redpanda Tiered Storage/Cloud Topics 工程文章、Bufstream Iceberg 文档镜像。
  • 交叉验证资料：TopicPartition / 2 Minute Streaming 对 Ursa 和 Lakestream 的第三方技术评论，以及本 workspace 既有 diskless_kafka_iceberg_lakehouse_survey_20260528.md。

注意：部分厂商横评来自竞品厂商，带有明显立场，仅用于发现维度，不直接作为事实依据。Bufstream 官方页面在检索时重定向到 CoreWeave acquisition 页面，因此 Bufstream 的 Iceberg 细节主要依赖其公开文档镜像与先前抓取结果，成熟度需要后续复核。

核心结论

  1. Ursa 最强的差异点不是 diskless，而是 diskless + leaderless + lakehouse-native。 WarpStream、AutoMQ、Aiven Inkless、Confluent Freight、Redpanda Cloud Topics 都在不同程度上把 Kafka log 推向对象存储；Ursa 更激进的地方是把 Kafka topic 直接变成 Iceberg/Delta table，把 streaming log 和 lakehouse table 视为同一份对象存储数据的两种视图。来源：PVLDB Ursa PDF、StreamNative Lakestream docs。

  2. Ursa 不是“更快的 Kafka”，而是“更便宜、更 lakehouse-native 的 Kafka-compatible ingest layer”。 论文明确把目标场景设在可容忍 sub-second latency，尤其 200-500ms 的数据密集 ingestion。它牺牲传统 Kafka sub-100ms 低延迟假设，换取跨 AZ 复制、本地磁盘和 Kafka-to-lakehouse connector 的成本下降。来源：PVLDB Ursa PDF。

  3. “leaderless”是 broker 层 leaderless，不是全系统无 leader。 Ursa 去掉的是 Kafka partition leader / ISR / broker local log 的中心性，但 offset assignment、ordering、transaction state、fencing、offset index、catalog commit 仍需要强一致 metadata/control plane。论文和 StreamNative 文档把 Oxia 放在 metadata layer；因此系统复杂性从 broker replication 转移到 Oxia、catalog、object storage request path、compaction service。来源：StreamNative Lakestream docs、PVLDB Ursa PDF。

  4. 最贴身竞品要分两张图看。 如果比较 diskless Kafka，最贴身是 WarpStream、AutoMQ、Aiven Inkless / Kafka KIP-1150、Confluent Freight、Redpanda Cloud Topics。如果比较 lakehouse-native streaming，最贴身是 Bufstream、AutoMQ Table Topic、WarpStream Tableflow、Confluent Tableflow、Aiven Iceberg Topics。Ursa 同时跨两张图，因此不能只用“Kafka 替代品”或“Kafka-to-Iceberg connector”来评价。

  5. 短期采购/落地时，Ursa 最大风险是成熟度和可验证性。 它有强论文和清晰架构，但 UFK 是 StreamNative 的 Kafka fork / managed service 路线；相比之下，Confluent/MSK/Kafka tiered storage 更保守成熟，WarpStream/AutoMQ 已有更明确的 diskless production narrative，Kafka KIP-1150 是上游方向但尚非完整生产实现。Ursa 的价值更像未来架构样板，落地前需要严格 POC。

  6. “零拷贝”需要逐产品拆开看。 Aiven Iceberg Topics/Bufstream Iceberg Archives/Ursa SBT 都使用“同一份对象存储数据服务 Kafka 和 Iceberg/table read”的叙事，但实现边界不同；Confluent Tableflow/WarpStream Tableflow 更明确是托管 materialization；AutoMQ Table Topic 介于 Kafka-native ingestion 和 table materialization 之间。报告中把“是否 zero-copy”视为需要验证的工程属性，而不是只看营销用词。

1. Ursa 到底是什么

1.1 它要解决的两个问题

Ursa 论文把问题拆成两个：

  • Kafka 在云上跑贵。 传统 Kafka 为 sub-100ms latency 设计，broker 本地磁盘 + leader/follower replication 在多 AZ 下产生高 cross-AZ 网络成本和多份磁盘成本。论文明确说很多现代 ingestion workload 只需要 sub-second latency，例如 200-500ms，此时继续使用传统 replication 架构成本过高。来源：PVLDB Ursa PDF。
  • Kafka 到 lakehouse 的路径太绕。 Kafka 数据通常要经过 Kafka Connect/Flink/Spark/ETL 再写 Iceberg/Delta/Hudi，产生两套系统、两份数据、延迟、schema/metadata 分裂和运维面。来源：PVLDB Ursa PDF。

所以 Ursa 的真正命题不是“Kafka 存储从 EBS 换成 S3”，而是：Kafka topic 能否原生成为 lakehouse table，同时仍保留 Kafka protocol / consumer semantics。

1.2 架构分层

StreamNative 的 Lakestream 文档把系统拆成三层：

  • Protocol Layer：Kafka/Pulsar brokers，stateless and leaderless，任意 broker 可处理 produce/fetch。
  • Metadata Layer：Oxia + catalog，管理强一致元数据、stream index、table metadata pointer。
  • Data Layer：Ursa Stream Storage，数据写 object storage，也可支持不同 WAL profile；数据以 Iceberg/Delta 等 open table format 暴露。来源：StreamNative Lakestream docs。

PVLDB 论文里的数据路径更具体：

  • WAL objects 聚合来自多个 stream/topic-partition 的新写入，优化 append。
  • Offset index 把 Kafka offset 映射到 WAL/Parquet 的位置。
  • Compaction service 把 row-based WAL objects 转换为 topic-partition-specific Parquet files。
  • Commit 阶段更新 Ursa metadata，并通过 Iceberg/Delta catalog commit 注册 Parquet 文件。
  • Kafka protocol server 对客户端保持 Kafka-compatible produce/fetch 语义。来源：PVLDB Ursa PDF。

这也是 Ursa 比普通 diskless Kafka 更有意思的地方：对象存储不是冷数据归档，而是 streaming log 和 lakehouse table 的共同底座。

1.3 UFK 的产品化说法

StreamNative 2026 年 UFK 文章说 Ursa-for-Kafka 是基于 Apache Kafka 4.2+ 的 fork，不是 proxy 或 protocol compatibility layer；它保留 Kafka request handlers、group coordinator、KRaft controller 等，同时把 local disk storage 扩展为 lakehouse-native storage option。来源：UFK Deep Dive。

另一个重要产品点是 mixed storage / dual profile：同一 Kafka fork 中可以保留传统 disk-based topic，也可以为部分 topic 启用 lakehouse-native/diskless profile。StreamNative 的 launch post 把这解释为迁移优势：先迁高吞吐、延迟不敏感、成本压力大的 topic，不需要 big-bang migration。来源：Ursa For Kafka launch post。

2. Ursa 的好处

2.1 成本结构：省 cross-AZ replication 和多份 broker disk

传统 Kafka 在多 AZ 下通常要跨 AZ 做 leader/follower replication，同时保留多份本地 segment。Ursa 把 durable replication 下沉到 object storage，broker 不再需要把每个 partition 的 log 复制到多个 broker 磁盘。对高吞吐、长保留、fanout 低或中等的 ingestion workload，这会直接改善成本结构。

这类收益不是 Ursa 独有，WarpStream、AutoMQ、Confluent Freight、Aiven KIP-1150 也都围绕同一个云成本结构展开。但 Ursa 把它和 table-native storage 合并，是差异点。

2.2 运维弹性：broker 更像 stateless serving layer

Ursa 的 broker 不拥有 partition local log，也不作为 partition leader 维持 ISR。扩容、缩容、broker failure 不需要传统 Kafka 那种 partition reassignment / leader election / local log catch-up。论文明确说 broker can be added or removed without rebalancing partitions or migrating data，失败后其他 broker 可通过 metadata + object storage 继续服务。来源：PVLDB Ursa PDF。

这是 diskless/leaderless 架构的共同吸引力：把最难操作的 Kafka broker state 从 broker 里拿掉。

2.3 Lakehouse-native：减少 Kafka-to-Iceberg 复制链路

Ursa 的 Stream-Backed Table 模式让 streaming index 和 lakehouse table metadata 指向同一份对象存储数据，外部引擎如 Spark/Trino/Flink 可读取 lakehouse table。论文还描述了 Stream-Delivered-to-Table 模式，可向 Unity Catalog、Snowflake Open Catalog 等外部 lakehouse table 交付。来源：PVLDB Ursa PDF。

这比“Kafka + connector + Iceberg sink”更像数据库系统意义上的 stream-table duality：stream 和 table 不是两个 pipeline stage，而是一份数据的两种访问语义。

2.4 迁移故事：保留 Kafka API 和部分 Kafka 内核

UFK 说自己是 Kafka fork，而不是重写 Kafka-compatible broker。这对采用者有两个潜在好处：

  • 客户端、consumer group、admin tooling、protocol surface 更容易继承 Kafka 生态。
  • 传统 topic 和 diskless/lakehouse-native topic 可分阶段混用。

但这个优势也带来 fork 风险，后面单独讨论。

3. Ursa 的缺点与隐性成本

3.1 它把 leader 从 broker 层移到 metadata/control plane

“leaderless”容易被误读。Ursa 确实去掉了 partition leader broker，但 metadata service 仍然要决定 offset、ordering、transaction visibility、fencing 和 index updates。Oxia 本身也需要强一致复制。也就是说，Ursa 不是消灭协调，而是把协调集中到更专门的 metadata layer。

这有合理性：数据库系统 disaggregation 本来就是把 log、metadata、storage 拆成专门服务。但评估成本时不能只看 broker 少了什么，还要看 Oxia quorum、catalog commit、metadata write amplification、metadata hot shard、跨 AZ placement 和 failover 行为。

3.2 延迟 trade-off 是架构内生的

对象存储 PUT/GET/list/range-read、WAL batching、metadata commit、Iceberg commit、compaction backlog 都会影响 tail latency。Ursa 论文把目标定位在 sub-second / 200-500ms tolerance，本身已经说明它不是传统低延迟 Kafka 的直接替代。

WarpStream 文档也给了类似证据：默认配置下 produce p99 约 500ms、E2E p99 约 900ms；使用 S3 Express 和更低 batch timeout 可降到 produce p99 <150ms，但会增加成本；Lightning Topics 可进一步降低 produce latency，但牺牲 full consistency。来源：WarpStream Low Latency Clusters。

所以 Ursa 的正确评价不是“能否比 Kafka 快”，而是“在可接受延迟下，能否显著降低总成本并减少 lakehouse pipeline”。

3.3 WAL 到 Parquet 的双层设计增加后台系统复杂度

Ursa 用 WAL objects 优化写入，用 Parquet compacted objects 优化 catch-up read 和 lakehouse query。这是合理设计，但意味着系统有两个持续运行的面：

  • hot path：produce/fetch、WAL、offset index、metadata commit。
  • background path：compaction、schema conversion、Parquet file sizing、Iceberg/Delta commit、WAL reclaim。

如果 compaction 落后，消费者 catch-up read 可能打到碎片化 WAL range reads；如果 compaction 太激进，则增加 object storage request cost 和 catalog commit 压力。这是所有 streaming-to-table 系统都会遇到的小文件/commit/freshness trade-off。

3.4 Kafka 语义和 Iceberg 语义不是天然等价

Kafka 关心 offset、ordering、consumer group、transaction、EOS、compacted topic、tombstone、retention。Iceberg/Delta 关心 snapshot、manifest、metadata pointer、schema evolution、delete files、compaction、table maintenance。

Ursa 的贡献正是试图用 offset index + WAL/Parquet + metadata commit 把两套语义桥起来。但难点也在这里。普通 append-only logging 容易成立；更难的 workload 包括：

  • compacted topic 和 tombstone。
  • transactional producers / exactly-once。
  • 高 fan-out catch-up readers。
  • key skew / hot partitions。
  • 高频 schema evolution。
  • 多租户下 catalog commit contention。
  • retention delete 和 Iceberg snapshot lifecycle 的交互。

论文和博客没有完全覆盖这些生产边界，所以 POC 必须围绕实际 workload，而不是只跑标准吞吐 benchmark。

3.5 Benchmark 和成本宣称需要拆账

Ursa、WarpStream、AutoMQ、Confluent、Redpanda 都有自己的成本/性能叙事。横向比较时必须统一口径：

  • 是否把 metadata service / control plane / coordinator 算入成本。
  • 是否把 compaction workers / table maintenance / catalog 服务算入成本。
  • 是否把 object storage PUT/GET/list/range request、S3 Express、cross-region、egress 算入成本。
  • 是否使用相同 retention、partition count、message size、producer concurrency、consumer fanout、catch-up workload。
  • 是否测试事务、compaction、schema evolution、故障恢复，而不是只测试 steady-state append。

如果不拆账，厂商的“10x cheaper”或“sub-10ms”很容易比较错对象。

5. 逐个竞品比较

5.1 WarpStream / Confluent WarpStream

WarpStream 是 Ursa 在 diskless Kafka 维度最直接的竞品之一。它的核心是 Kafka-compatible stateless agents，无本地磁盘，直接写 object storage，并把 metadata 放到 WarpStream Cloud Metadata Store；官方架构文档说任意 Agent 都可以 act as leader for any topic、commit offsets、act as coordinator，扩缩容像运行 proxy/web server。来源：WarpStream architecture。WarpStream 自己承认 object storage write latency 比本地磁盘高，并在 tuning 文档里用 Little's Law 解释：要用高并发和大 batch 才能吃满吞吐；单个 4MiB S3 write p99 可到约 400ms。来源：WarpStream tuning docs。

WarpStream write path 的关键细节是：producer 尽量连同 AZ Agent；任意 Agent 可写任意 topic-partition；数据必须先 durably persisted in object storage 并 committed to metadata store，才 ack produce request；默认 buffer 250ms 或 8MiB。来源：WarpStream write path。这和 Ursa 的 WAL/object + metadata commit 模式高度相似，但 WarpStream 的输出首先是 Kafka-compatible object log，而不是原生 Iceberg/Delta table。

WarpStream 的低延迟策略包括降低 agent batch timeout、使用更高 control plane tier、使用 S3 Express、使用 Lightning Topics。官方表格显示默认 S3 Standard profile 下 produce p99 约 500ms、E2E p99 约 900ms；S3 Express + 50ms batch timeout 可到 produce p99 <150ms、E2E p99 <400ms；Lightning Topics produce p99 <50ms，但不是 full consistency。来源：WarpStream Low Latency Clusters。Tableflow 方面，WarpStream 文档说它把 Kafka-compatible topic 转成 Iceberg table，BYOC 下 compute/storage 在用户 VPC，table raw data 只存在用户 object storage，compaction/table maintenance 内置；但 Tableflow agents 不暴露 Kafka protocol，是独立 table materialization cluster。来源：WarpStream Tableflow setup。

和 Ursa 的关键区别：

  • WarpStream 的主线是 diskless Kafka-compatible log，lakehouse 是 Tableflow materialization。
  • Ursa 的主线是 lakehouse-native storage engine，topic/table 是更底层的统一对象。
  • WarpStream 的生产采用叙事更成熟；Ursa 的 stream-table duality 更激进。

如果你要比较“谁更适合替换高成本 Kafka ingestion cluster”，WarpStream 是 Ursa 的一线竞品。如果你要比较“谁把 topic 原生变成 Iceberg/Delta”，WarpStream Tableflow 更像 Confluent Tableflow，不完全是同构竞品。

5.2 AutoMQ

AutoMQ 自称 diskless Kafka on S3、100% Kafka compatible、sub-10ms P99 latency、zero cross-AZ costs。来源：AutoMQ homepage、AutoMQ GitHub。它和 Ursa 一样抓住云上 Kafka 的核心成本问题，但技术取舍不同。

AutoMQ 文档的 performance benchmark 明确提示：AutoMQ BYOC 使用 EBS as WAL storage 来实现低延迟 send/receive；也就是说，它不是简单“所有 hot path 直接写 S3”的模型，而是用 WAL + shared/object storage 分层换取延迟。来源：AutoMQ Performance Benchmark。这解释了它为什么能讲 sub-10ms：低延迟 profile 的 durable write path 和 WarpStream/Ursa 的对象存储 path 不是完全同一种比较对象。

AutoMQ 也在补 lakehouse-native 能力。Table Topic 文档说它可把 streaming data ingest into Iceberg tables，内置 schema registry，自动创建 Iceberg table，支持 AWS Glue / S3 Table Catalog，并减少 Kafka Connect/Flink 这类 ETL pipeline。来源：AutoMQ Table Topic。

和 Ursa 的关键区别：

  • AutoMQ 更强调 Kafka-compatible、低延迟、弹性和成本；Ursa 更强调 Kafka topic 和 lakehouse table 的统一存储语义。
  • AutoMQ Table Topic 看起来接近 Ursa 的 lakehouse 目标，但是否是严格 zero-copy、Kafka fetch 与 Iceberg query 是否共享同一批 Parquet objects，需要按版本和配置验证。
  • AutoMQ 的跨 AZ 优化有前提，例如三 AZ、producer/consumer 和 AutoMQ balanced AZ deployment；文档也说不满足条件会产生部分 cross-AZ traffic。来源：AutoMQ broker configuration。

AutoMQ 是 Ursa 在“Kafka-compatible diskless + lakehouse feature”两条线上都值得认真比较的竞品。

5.3 Bufstream

Bufstream 是 Ursa 在 lakehouse-native 维度最贴身的竞品之一。公开资料描述它是 Kafka-compatible broker / replacement，使用 object storage，并把 topic data 以 Iceberg archives 方式写成 Parquet + Iceberg metadata；Parquet files 同时作为 Kafka consumer 的数据和 Iceberg table 的 data files。先前检索到的 Bufstream docs 明确称 Iceberg Archives 是 zero copy，不为 Iceberg 另建一份数据。来源：Bufstream Iceberg Archives reference。

Bufstream 的独特点是 schema/data quality：它依赖 Protobuf schema registry，把 broker-side semantic validation 和 Protobuf-to-Iceberg schema mapping 放进写入链路。来源：Bufstream Iceberg configuration。

和 Ursa 的关键区别：

  • Bufstream 更像“Kafka-compatible + Protobuf/schema-first + Iceberg storage”的产品路线。
  • Ursa 更像“Kafka/Pulsar protocol + Lakestream storage layer + Oxia metadata + Iceberg/Delta”的架构路线。
  • Bufstream 与 Ursa 都值得用 Kafka semantics + Iceberg semantics 的一致性来压测，尤其 transaction、compacted topic、schema evolution、invalid records、retention/snapshot lifecycle。

由于 Bufstream 页面在当前检索中有重定向，产品路线和可获得性需要另行确认；但在架构比较上，它是最重要的 lakehouse-native 对照组。

5.4 Aiven Iceberg Topics

Aiven Iceberg Topics 是另一条很重要但容易被忽略的路线：它不是另起一个 Kafka-compatible broker，而是从 Kafka tiered storage / Remote Storage Manager 插件方向，把 topic segment 直接持久化为 Parquet/Iceberg，让同一份 durable copy 同时服务 Kafka replay 和 Iceberg analytics。前一份 survey 中记录的 whitepaper 表述是：segment rotate 时，RemoteStorageManager 将 records persist as Parquet files into an Iceberg table；fetch 时再 reconstruct valid Kafka batches。来源：Aiven Iceberg whitepaper。

这条路线和 Ursa 的区别是：

  • Aiven 更贴近 Apache Kafka 内部扩展点，vendor/fork 风险较小。
  • Ursa 是更完整的外部 storage engine + metadata layer 设计，架构控制力更强。
  • Aiven Iceberg Topics 的优势是 Kafka-native；弱点是成熟度和能力边界。前一份报告记录其 README 曾标注 Iceberg mode recently added / alpha stage。来源：Aiven tiered-storage README。

如果要比较“Kafka 上游/插件体系能不能自然长出 topic-as-table”，Aiven Iceberg Topics 是 Ursa 的关键参照。

5.5 Apache Kafka KIP-1150 / Aiven Inkless

KIP-1150 是 Ursa 的长期标准化参照。Aiven 2026 年 3 月文章说 KIP-1150 Diskless Topics 已被 Apache Kafka 社区接受；它把 KIP-1150 定位为 meta/motivational KIP，先确认“Kafka 是否要 diskless topics”，具体实现拆到后续子 KIP，如 Diskless Core 和 Diskless Coordinator。来源：Aiven KIP-1150 accepted。Aiven 的 Inkless 是这个方向的产品化/实验实现；Aiven 博客把动机描述为更快升级和恢复、降低 infra/network 成本，并称 Inkless clusters 已在 Aiven Cloud 提供。来源：Aiven Inkless announcement、Aiven Diskless Kafka。

KIP-1150 的方向和 Ursa 相似：diskless topic 不用 broker disk 作为 user data 的 primary durable storage，Kafka 把 replication 交给 object storage，broker 不再唯一作为 partition leader。KIP-1163 进一步定义 produce/fetch path，KIP-1164 负责 diskless coordinator，KIP-1183 则提出 unified shared storage / shared log 抽象。来源：KIP-1150、KIP-1163、KIP-1164、KIP-1183。

和 Ursa 的关键区别：

  • KIP-1150 是 Apache Kafka 上游方向，不是完整产品。
  • 它主要解决 diskless topics，不内建 Ursa 式 Iceberg/Delta table duality。
  • 如果 KIP-1150 成熟，Ursa 的 Kafka fork 差异化会被部分压缩；但 Ursa 的 lakehouse-native/cross-protocol Lakestream 仍可能保持差异。
  • Aiven 公开 benchmark 给出了 diskless topic 的现实延迟/成本轮廓；这些数字可以作为 Ursa POC 的参照，但不应和 Ursa 的 lakehouse-native table path 直接混比。来源：Aiven benchmarking diskless topics。

如果你的核心诉求是“跟随 Kafka 上游，降低长期 fork/vendor risk”，KIP-1150/Aiven Inkless 是重要参照。如果核心诉求是“现在就把 topic 变 lakehouse table”，它还不是直接替代。

5.6 Confluent Freight + Tableflow

Confluent 是保守企业路径的代表。Freight clusters 把数据 direct write 到 object storage，避免 Kora brokers 的 local storage 和 replication；Confluent 自己把它定位为低成本、延迟可放宽的 cluster type，延迟可到 up to a second or two。来源：Confluent Freight GA。

Tableflow 则把 Kafka topic materialize 为 Iceberg/Delta table。Confluent docs 把它定义为 Kafka topics as Iceberg and Delta Lake tables；GA blog 说它会把 Kafka messages 转成 Parquet，组织成 Iceberg tables，并管理 compaction 等 metadata/maintenance。来源：Confluent Tableflow docs、Tableflow GA blog。

和 Ursa 的关键区别：

  • Confluent 路线把 “Kafka storage” 和 “Kafka-to-table materialization” 拆成 Freight + Tableflow 两个产品面。
  • Ursa 把这两件事合并到一个 lakehouse-native storage engine 里。
  • Confluent 成熟度、SLA、生态、治理工具强；Ursa 架构更干净但更新、更激进。

现实采购中，Confluent 可能是 Ursa 最常遇到的保守替代：不是同构，但对企业用户来说是可接受风险最低的方案。

5.7 Redpanda Tiered Storage / Cloud Topics

Redpanda 不是 Ursa 式 lakehouse-native 系统，但它是 Kafka API-compatible 替代和 object-storage trend 的重要参照。Redpanda Tiered Storage 文档说它把 log segments offload 到 object storage，降低存储成本，并通过 remote write / remote read 访问远端数据。来源：Redpanda Tiered Storage。

Redpanda 2026 年 Cloud Topics 工程文章重点讨论 object storage 上 streaming 的两个实际问题：Little's Law 下高延迟存储如何靠并发维持吞吐，以及 Level Zero temporary objects 的生命周期/垃圾回收。来源：Little's Law in practice with Cloud Topics、Cloud Topics: Level Zero garbage collection。

和 Ursa 的关键区别：

  • Redpanda 优先级是 high-performance Kafka-compatible streaming，不是 topic-as-Iceberg-table。
  • 它适合和 Ursa 比较低延迟、运维、Kafka compatibility、tiered/object storage 成熟度。
  • 如果 workload 仍然是 operational streaming，而不是 lakehouse ingestion，Redpanda 可能比 Ursa 更自然。

5.8 Fluss

Fluss 提供的是另一种反方参照：它不主张直接让 Iceberg 承担全部 hot streaming storage，而是把 Iceberg/Paimon 视为 lakehouse/cold tier，把低延迟 streaming、primary-key semantics、实时更新留给专门的 hot streaming storage。前一份 survey 已记录它的判断：Iceberg 更适合作为 cold tier，Fluss 提供 hot streaming storage、sub-second latency、primary-key semantics，然后 tiering 到 Iceberg/Paimon。来源：Fluss x Iceberg。

它不是 Ursa 的直接采购竞品，但对技术判断很重要：Ursa 的隐含判断是“streaming log 和 lakehouse table 可以足够深地统一”；Fluss 的判断是“hot stream 和 lakehouse table 应该分层，只在边界上衔接”。如果 workload 有高频更新、低延迟 primary-key read/update、复杂 changelog，Fluss 这类分层路线可能比 Ursa 式 topic-as-table 更自然。

6. 最适合的 workload 与不适合的 workload

6.1 Ursa 适合

  • 高吞吐日志、metrics、clickstream、IoT ingestion。
  • CDC / data lake ingestion。
  • AI/ML pipeline 的原始事件落湖。
  • 需要较长 retention，且希望减少 Kafka 本地磁盘和 connector pipeline 的场景。
  • 可容忍 200ms-秒级 freshness，而不是 sub-50ms tail latency 的场景。
  • 希望让 Spark/Trino/Flink/Snowflake/Databricks 直接读 streaming data 的场景。

6.2 Ursa 不适合

  • 低延迟交易、在线风控、request-response event bus。
  • 严重依赖 Kafka compaction / tombstone / transaction / EOS 的复杂 workload，除非 POC 充分覆盖。
  • 小消息、低并发、强 tail-latency SLO 的 workload。
  • 不愿承担新 metadata/catalog control plane 的团队。
  • 已有成熟 Kafka + Connect/Flink + Iceberg pipeline，且成本不是主要瓶颈的团队。

7. 如果要 POC Ursa，应该怎么测

7.1 性能与成本

  • 用真实 message size、partition count、producer concurrency、consumer fanout，而不是只跑标准 OpenMessaging happy path。
  • 分别测 p50/p95/p99/p999 produce latency、E2E latency、catch-up read latency。
  • 测低并发小 batch 和高并发大 batch，因为 object storage 系统对 Little's Law 非常敏感。
  • 成本必须拆出 broker、Oxia/metadata、compaction workers、catalog、object storage PUT/GET/list/range requests、S3 Express、cross-AZ/cross-region、egress。

7.2 Kafka 语义

  • transactional producer / exactly-once。
  • consumer group churn、rebalance、offset commit。
  • compacted topics、tombstone、retention delete。
  • key skew、hot partition、多租户隔离。
  • broker crash、metadata leader failover、object storage throttling、AZ failure。

7.3 Lakehouse 语义

  • Iceberg/Delta table freshness SLA。
  • 小文件数量、compaction lag、snapshot count、manifest growth。
  • schema evolution，高频 schema 变更，invalid records / DLQ。
  • 外部 query engine 读一致性：Spark/Trino/Snowflake/Databricks 是否看到预期 snapshot。
  • catalog lock contention 和 catalog vendor lock-in。

8. 判断框架

如果目标是 降 Kafka 成本，但仍主要服务 Kafka consumers，优先比较 WarpStream、AutoMQ、Confluent Freight、Aiven/KIP-1150、Redpanda Cloud Topics。Ursa 可以纳入，但不要只因 lakehouse-native 加分。

如果目标是 把 Kafka ingestion 直接变成 Iceberg/Delta 数据资产，优先比较 Ursa、Bufstream、AutoMQ Table Topic、WarpStream Tableflow、Confluent Tableflow、Aiven Iceberg Topics。这里 Ursa 是最有架构野心的选项之一。

如果目标是 企业稳妥落地，Confluent Cloud/MSK/Kafka + Tableflow/Connect/Flink 仍是现实 baseline。它可能不够优雅，也可能更贵，但 maturity、SLA、生态、招聘/运维熟悉度更强。

如果目标是 长期技术方向判断，KIP-1150/1163/1164 比任何单一厂商更重要。上游 Kafka 一旦形成 diskless topic 标准，vendor differentiation 会从“能不能 diskless”转向“metadata、catalog、maintenance、governance、lakehouse integration 做得多好”。

9. Ursa Bear Case：为什么它可能不适合你

如果只读 StreamNative 的论文和 launch posts，Ursa 很容易显得像一个同时解决 Kafka 成本、broker 运维、lakehouse ingestion、stream-table duality 的统一答案。但反方视角看，它至少有七个可能翻车点。

9.1 它可能只是把 Kafka 的复杂性换了位置

Ursa 去掉了 Kafka broker 上的 partition leader、ISR、本地 log 和 rebalancing，但没有去掉 ordering、offset assignment、transaction visibility、fencing、metadata commit 这些硬问题。它们被移到 Oxia、offset index、catalog commit 和 compaction service。

这意味着 Ursa 的运维问题不再是“哪个 broker 磁盘满了、哪个 partition leader 热了”，而变成：

  • Oxia/metadata shard 是否成为 hot partition。
  • metadata quorum 跨 AZ 成本和延迟是否被充分计入。
  • catalog commit 是否成为高频写入下的新瓶颈。
  • offset index 和 table metadata 是否能在故障恢复时保持一致。
  • compaction service 落后时，Kafka replay 和 table query 是否一起劣化。

所以 Ursa 的复杂性不是消失，而是从熟悉的 Kafka broker 层转移到了较新的 metadata/table-control-plane 层。对团队来说，这可能是降维，也可能是换成一套更陌生的故障模式。

9.2 它的最佳场景比营销叙事窄

Ursa 最适合的是高吞吐、可容忍 sub-second freshness、以落湖/分析/AI ingestion 为核心的数据流。如果 workload 主要是 operational streaming，比如低延迟事件总线、订单状态流、风控信号、在线服务协调、强 p99/p999 SLO，那么 Ursa 的 lakehouse-native 优势可能用不上，object storage 和 metadata commit 的代价反而会暴露。

换句话说，Ursa 的问题不是“能不能跑 Kafka API”，而是“你的 Kafka workload 是否真的愿意用 200-500ms 或更高 tail latency 换成本和 lakehouse 可见性”。如果答案是否定的，Redpanda、传统 Kafka、Confluent low-latency profile、AutoMQ 低延迟 profile 或 WarpStream S3 Express profile 可能更合理。

9.3 Topic-as-table 可能低估了 Iceberg/Delta 的维护成本

“每个 Kafka topic 都是 Iceberg/Delta table”听起来很干净，但生产 Iceberg/Delta 表不是只写 Parquet 文件。它还需要处理：

  • 小文件合并和 compaction backlog。
  • snapshot expiration 和 orphan file cleanup。
  • schema evolution 与坏数据处理。
  • partition/sort strategy 对查询性能的影响。
  • catalog commit contention。
  • table maintenance 与 Kafka retention 的语义边界。

传统 Kafka-to-Iceberg pipeline 的复杂性虽然丑，但至少是显式的：Connect/Flink/Spark job、DLQ、catalog、compaction、monitoring 各自可见。Ursa 把这条 pipeline 内生化以后，运维面更少，但系统内部必须把这些事情做对。对使用者来说，这可能降低运维负担，也可能降低可观察性和可控性。

9.4 Kafka compatibility 的难点在长尾语义，不在 happy path

Produce/fetch、consumer group、普通 append-only topic 并不能证明 Ursa 等价于 Kafka。真正需要压测的是长尾语义：

  • transactions / exactly-once。
  • compacted topic / tombstone。
  • key skew 和 hot partition。
  • high fan-out catch-up readers。
  • retention delete 与 Iceberg snapshot lifecycle 的交互。
  • consumer group churn 和 offset commit 风暴。
  • schema evolution storm。
  • object storage throttling、catalog outage、metadata failover 下的恢复。

这些场景也是 vendor benchmark 最容易覆盖不足的地方。如果 POC 只跑吞吐和平均延迟，Ursa 的风险会被系统性低估。

9.5 成本优势可能被 control plane 和 table maintenance 吃掉一部分

Ursa 的成本节省主要来自减少 cross-AZ broker replication、本地磁盘和 connector pipeline。但真实账单还要加上：

  • Oxia/metadata service 机器、存储、quorum traffic。
  • Compaction workers。
  • Object storage PUT/GET/range request/list cost。
  • Iceberg/Delta catalog 和 table maintenance。
  • Query engine 读取未优化小文件时的放大。
  • 为降低延迟而使用更贵存储层或更小 batch 的成本。

这不意味着 Ursa 不省钱，而是说“10x cheaper”这类结论必须在自己的 workload 上拆账。尤其当 topic 数量多、message 小、schema 变化频繁、查询 freshness 要求高时，metadata 和 maintenance 成本会比营销材料里更显眼。

9.6 Fork 路线有长期维护风险

UFK 的一个卖点是它是 Kafka fork，不是 proxy。但 fork 同时也是风险：

  • Kafka 上游继续演进 KIP-1150/1163/1164 后，UFK 如何跟进。
  • Kafka 安全补丁、protocol 变化、group coordinator/KRaft 行为变化如何同步。
  • 周边工具如果依赖 broker 内部行为，而非纯 protocol，是否有兼容性差异。
  • 如果未来 Apache Kafka 原生 diskless topics 成熟，UFK 的差异化是否转向 proprietary storage/catalog layer。

这类风险短期不一定影响 POC，但会影响 3-5 年平台选择。

9.7 最大的不确定性是成熟度，而不是架构是否漂亮

Ursa 的架构方向很有说服力，也有 PVLDB 论文加持。但对平台团队来说，问题不是“这套架构是否先进”，而是：

  • 是否有足够多公开生产案例。
  • 是否能观测和调试 Oxia、offset index、compaction、catalog commit。
  • 是否能给出清晰的故障恢复 runbook。
  • 是否支持你依赖的 Kafka 语义和治理工具。
  • 是否能在你的云、你的对象存储、你的 catalog、你的查询引擎组合里稳定运行。

因此，Ursa 更适合作为高潜力候选进入 POC，而不是直接作为 Kafka 平台默认替代。最安全的采用方式是先选高吞吐、低延迟敏感度、append-heavy、落湖价值高的 topic，而不是迁移核心 operational topics。

10. 最终结论

Ursa 值得重点研究，因为它把三条趋势压到了一起：

  • Kafka broker stateless / diskless。
  • Cloud object storage 作为 durable streaming log。
  • Iceberg/Delta catalog 作为 streaming data 的 table control plane。

它的好处是清晰的：省跨 AZ replication 和本地磁盘，简化 broker 运维，把 Kafka-to-lakehouse pipeline 变成原生存储路径。它的风险也同样清晰：metadata/control plane 变成新 hot path，object storage latency 和 request cost 进入系统设计，Iceberg/Delta maintenance 不再是旁路问题，Kafka fork/产品成熟度需要验证。

最准确的评价是：Ursa 是 lakehouse-native streaming engine 的强样板，不是所有 Kafka workload 的通用替代。 对高吞吐、成本敏感、分析/AI/CDC 落湖场景，它是非常有吸引力的方向；对低延迟 operational streaming，传统 Kafka/Redpanda 或低延迟 profile 的 AutoMQ/Confluent/WarpStream 可能更合适。

更尖锐地说：Ursa 的最大价值是把 Kafka 变成 lakehouse ingestion/control-plane 的一部分；它的最大风险也是这一点。你得到的不只是“便宜 Kafka”，而是一套新的 streaming-table storage engine。只有当 lakehouse-native 的收益足以覆盖 metadata、catalog、compaction、fork 和成熟度风险时，Ursa 才是优选。

下一步如果要继续深入，最有价值的是做一份 POC checklist 或 benchmark matrix，把 Ursa、WarpStream、AutoMQ、Bufstream/Confluent Tableflow 放到同一套 workload 下测，而不是继续比较厂商宣称。

Core Conclusions

  1. Ursa's strongest differentiator is not diskless alone, but diskless + leaderless + lakehouse-native. WarpStream, AutoMQ, Aiven Inkless, Confluent Freight, and Redpanda Cloud Topics all move the Kafka log toward object storage to varying degrees. Ursa is more aggressive because it turns a Kafka topic directly into an Iceberg/Delta table, treating the streaming log and the lakehouse table as two views over the same object storage data. Sources: PVLDB Ursa PDF, StreamNative Lakestream docs.

  2. Ursa is not "faster Kafka", but a cheaper, more lakehouse-native Kafka-compatible ingest layer. The paper explicitly targets workloads that can tolerate sub-second latency, especially data-intensive ingestion in the 200-500ms range. It trades away the traditional Kafka sub-100ms low-latency assumption in exchange for lower costs from cross-AZ replication, local disk, and Kafka-to-lakehouse connectors. Source: PVLDB Ursa PDF.

  3. "Leaderless" means leaderless at the broker layer, not leaderless across the whole system. Ursa removes the centrality of the Kafka partition leader, ISR, and broker local log, but offset assignment, ordering, transaction state, fencing, offset indexes, and catalog commits still require a strongly consistent metadata/control plane. The paper and StreamNative docs place Oxia in the metadata layer, so system complexity moves from broker replication to Oxia, the catalog, the object storage request path, and the compaction service. Sources: StreamNative Lakestream docs, PVLDB Ursa PDF.

  4. The closest competitors need to be shown on two maps. For diskless Kafka, the closest competitors are WarpStream, AutoMQ, Aiven Inkless / Kafka KIP-1150, Confluent Freight, and Redpanda Cloud Topics. For lakehouse-native streaming, the closest competitors are Bufstream, AutoMQ Table Topic, WarpStream Tableflow, Confluent Tableflow, and Aiven Iceberg Topics. Ursa spans both maps, so it cannot be evaluated only as a "Kafka replacement" or a "Kafka-to-Iceberg connector".

  5. For near-term procurement or deployment, Ursa's biggest risks are maturity and verifiability. It has a strong paper and a clear architecture, but UFK follows StreamNative's Kafka fork / managed service path. By contrast, Confluent/MSK/Kafka tiered storage is more conservative and mature, WarpStream/AutoMQ have clearer diskless production narratives, and Kafka KIP-1150 is the upstream direction but not yet a complete production implementation. Ursa's value is closer to a future architecture template, and deployment requires a strict POC.

  6. "Zero-copy" has to be decomposed product by product. Aiven Iceberg Topics, Bufstream Iceberg Archives, and Ursa SBT all use a narrative where the same object storage data serves Kafka and Iceberg/table reads, but their implementation boundaries differ. Confluent Tableflow and WarpStream Tableflow are more clearly managed materialization paths. AutoMQ Table Topic sits between Kafka-native ingestion and table materialization. This report treats "whether it is zero-copy" as an engineering property that must be verified, not as a marketing phrase.

1. What Ursa Actually Is

1.1 The Two Problems It Tries to Solve

The Ursa paper splits the problem into two parts:

  • Kafka is expensive to run in the cloud. Traditional Kafka is designed for sub-100ms latency. Broker local disk plus leader/follower replication creates high cross-AZ network costs and multiple copies of disk data in multi-AZ deployments. The paper explicitly says many modern ingestion workloads only need sub-second latency, such as 200-500ms, making the traditional replication architecture too expensive. Source: PVLDB Ursa PDF.
  • The path from Kafka to the lakehouse is too indirect. Kafka data usually passes through Kafka Connect/Flink/Spark/ETL before being written to Iceberg/Delta/Hudi, creating two systems, two copies of data, added latency, split schema/metadata, and extra operational surface. Source: PVLDB Ursa PDF.

So Ursa's real claim is not "replace EBS with S3 for Kafka storage". It is: can a Kafka topic natively become a lakehouse table while still preserving Kafka protocol / consumer semantics?

1.2 Architecture Layers

StreamNative's Lakestream docs split the system into three layers:

  • Protocol Layer: Kafka/Pulsar brokers, stateless and leaderless, where any broker can handle produce/fetch.
  • Metadata Layer: Oxia + catalog, managing strongly consistent metadata, stream indexes, and table metadata pointers.
  • Data Layer: Ursa Stream Storage, where data is written to object storage and can also support different WAL profiles. Data is exposed through open table formats such as Iceberg/Delta. Source: StreamNative Lakestream docs.

The data path in the PVLDB paper is more specific:

  • WAL objects aggregate new writes from multiple streams/topic-partitions to optimize append.
  • The offset index maps Kafka offsets to locations in WAL/Parquet.
  • The compaction service converts row-based WAL objects into topic-partition-specific Parquet files.
  • The commit phase updates Ursa metadata and registers Parquet files through Iceberg/Delta catalog commits.
  • The Kafka protocol server preserves Kafka-compatible produce/fetch semantics for clients. Source: PVLDB Ursa PDF.

This is also why Ursa is more interesting than ordinary diskless Kafka: object storage is not a cold-data archive, but the shared foundation for the streaming log and the lakehouse table.

1.3 UFK's Product Positioning

StreamNative's 2026 UFK post says Ursa-for-Kafka is a fork based on Apache Kafka 4.2+, not a proxy or protocol compatibility layer. It keeps Kafka request handlers, the group coordinator, the KRaft controller, and so on, while extending local disk storage into a lakehouse-native storage option. Source: UFK Deep Dive.

Another important product point is mixed storage / dual profile: the same Kafka fork can keep traditional disk-based topics while enabling a lakehouse-native/diskless profile for selected topics. StreamNative's launch post frames this as a migration advantage: start with high-throughput, latency-insensitive, cost-pressured topics without a big-bang migration. Source: Ursa For Kafka launch post.

2. Benefits of Ursa

2.1 Cost Structure: Lower Cross-AZ Replication and Fewer Broker Disk Copies

Traditional Kafka in multi-AZ deployments usually performs leader/follower replication across AZs while keeping multiple local segment copies. Ursa pushes durable replication down into object storage, so brokers no longer need to replicate each partition log to multiple broker disks. For high-throughput, long-retention ingestion workloads with low or moderate fanout, this directly improves the cost structure.

This benefit is not unique to Ursa. WarpStream, AutoMQ, Confluent Freight, and Aiven KIP-1150 are all aimed at the same cloud cost structure. Ursa's differentiator is that it combines this with table-native storage.

2.2 Operational Elasticity: Brokers Become More Like a Stateless Serving Layer

Ursa brokers do not own partition local logs and do not maintain ISR as partition leaders. Scale-out, scale-in, and broker failure do not require traditional Kafka partition reassignment / leader election / local log catch-up. The paper explicitly says brokers can be added or removed without rebalancing partitions or migrating data, and after failures, other brokers can continue serving through metadata + object storage. Source: PVLDB Ursa PDF.

This is the shared appeal of diskless/leaderless architecture: remove the hardest-to-operate Kafka broker state from the broker.

2.3 Lakehouse-Native: Reducing the Kafka-to-Iceberg Copy Chain

Ursa's Stream-Backed Table mode lets the streaming index and lakehouse table metadata point to the same object storage data, so external engines such as Spark/Trino/Flink can read the lakehouse table. The paper also describes Stream-Delivered-to-Table mode, which can deliver data into external lakehouse tables such as Unity Catalog and Snowflake Open Catalog. Source: PVLDB Ursa PDF.

This looks more like stream-table duality in the database systems sense than "Kafka + connector + Iceberg sink": stream and table are not two pipeline stages, but two access semantics over one copy of data.

2.4 Migration Story: Keeping the Kafka API and Part of the Kafka Kernel

UFK says it is a Kafka fork rather than a rewritten Kafka-compatible broker. This gives adopters two potential benefits:

  • Clients, consumer groups, admin tooling, and the protocol surface are more likely to inherit the Kafka ecosystem.
  • Traditional topics and diskless/lakehouse-native topics can be mixed in phases.

But this same advantage also creates fork risk, discussed below.

3. Ursa's Weaknesses and Hidden Costs

3.1 It Moves the Leader from the Broker Layer to the Metadata/Control Plane

"Leaderless" is easy to misread. Ursa does remove the partition leader broker, but the metadata service still has to decide offsets, ordering, transaction visibility, fencing, and index updates. Oxia itself also requires strongly consistent replication. In other words, Ursa does not eliminate coordination. It concentrates coordination in a more specialized metadata layer.

This is reasonable: database system disaggregation is precisely about splitting log, metadata, and storage into specialized services. But cost evaluation cannot only look at what brokers no longer do. It must also account for Oxia quorum, catalog commits, metadata write amplification, metadata hot shards, cross-AZ placement, and failover behavior.

3.2 The Latency Trade-off Is Built Into the Architecture

Object storage PUT/GET/list/range-read, WAL batching, metadata commits, Iceberg commits, and compaction backlog all affect tail latency. The Ursa paper targets sub-second / 200-500ms tolerance, which already signals that it is not a direct replacement for traditional low-latency Kafka.

WarpStream's docs provide similar evidence: under default configuration, produce p99 is around 500ms and E2E p99 around 900ms. With S3 Express and a lower batch timeout, produce p99 can fall below 150ms, but cost increases. Lightning Topics can reduce produce latency further, but give up full consistency. Source: WarpStream Low Latency Clusters.

So the right question for Ursa is not "can it be faster than Kafka", but "under acceptable latency, can it meaningfully lower total cost and reduce the lakehouse pipeline".

3.3 The WAL-to-Parquet Two-Layer Design Adds Background System Complexity

Ursa uses WAL objects to optimize writes and Parquet compacted objects to optimize catch-up reads and lakehouse queries. This is a reasonable design, but it means the system has two continuously running planes:

  • hot path: produce/fetch, WAL, offset index, metadata commit.
  • background path: compaction, schema conversion, Parquet file sizing, Iceberg/Delta commit, WAL reclaim.

If compaction falls behind, consumer catch-up reads may hit fragmented WAL range reads. If compaction is too aggressive, object storage request cost and catalog commit pressure increase. This is the small-file/commit/freshness trade-off that every streaming-to-table system faces.

3.4 Kafka Semantics and Iceberg Semantics Are Not Naturally Equivalent

Kafka cares about offsets, ordering, consumer groups, transactions, EOS, compacted topics, tombstones, and retention. Iceberg/Delta cares about snapshots, manifests, metadata pointers, schema evolution, delete files, compaction, and table maintenance.

Ursa's contribution is exactly the attempt to bridge these two semantic systems with offset indexes + WAL/Parquet + metadata commits. But the hard part is also here. Ordinary append-only logging is easier. More difficult workloads include:

  • compacted topics and tombstones.
  • transactional producers / exactly-once.
  • high fan-out catch-up readers.
  • key skew / hot partitions.
  • high-frequency schema evolution.
  • catalog commit contention under multi-tenancy.
  • interactions between retention deletes and the Iceberg snapshot lifecycle.

The paper and blog posts do not fully cover these production boundaries, so a POC must be built around the actual workload rather than only running standard throughput benchmarks.

3.5 Benchmark and Cost Claims Need Itemized Accounting

Ursa, WarpStream, AutoMQ, Confluent, and Redpanda each have their own cost/performance narrative. Cross-product comparison requires a unified accounting basis:

  • whether metadata service / control plane / coordinator cost is included.
  • whether compaction workers / table maintenance / catalog services are included.
  • whether object storage PUT/GET/list/range requests, S3 Express, cross-region, and egress costs are included.
  • whether the same retention, partition count, message size, producer concurrency, consumer fanout, and catch-up workload are used.
  • whether transactions, compaction, schema evolution, and failure recovery are tested, not only steady-state append.

Without itemized accounting, vendor claims such as "10x cheaper" or "sub-10ms" are easy to compare against the wrong target.

5. Competitor-by-Competitor Comparison

5.1 WarpStream / Confluent WarpStream

WarpStream is one of Ursa's most direct competitors on the diskless Kafka dimension. Its core is Kafka-compatible stateless agents with no local disk, direct writes to object storage, and metadata in the WarpStream Cloud Metadata Store. The official architecture docs say any Agent can act as leader for any topic, commit offsets, and act as coordinator, while scaling resembles running a proxy/web server. Source: WarpStream architecture. WarpStream itself acknowledges that object storage write latency is higher than local disk, and its tuning docs use Little's Law to explain that high concurrency and large batches are needed to saturate throughput. A single 4MiB S3 write can have p99 around 400ms. Source: WarpStream tuning docs.

The key details of WarpStream's write path are: the producer tries to connect to an Agent in the same AZ, any Agent can write any topic-partition, and data must first be durably persisted in object storage and committed to the metadata store before the produce request is acknowledged. The default buffer is 250ms or 8MiB. Source: WarpStream write path. This is highly similar to Ursa's WAL/object + metadata commit pattern, but WarpStream's output is first a Kafka-compatible object log, not a native Iceberg/Delta table.

WarpStream's low-latency strategies include lowering the agent batch timeout, using a higher control plane tier, using S3 Express, and using Lightning Topics. The official table shows produce p99 around 500ms and E2E p99 around 900ms under the default S3 Standard profile. S3 Express + 50ms batch timeout can reach produce p99 <150ms and E2E p99 <400ms. Lightning Topics can reach produce p99 <50ms, but not full consistency. Source: WarpStream Low Latency Clusters. On Tableflow, WarpStream docs say it converts Kafka-compatible topics into Iceberg tables. Under BYOC, compute/storage run in the user's VPC, table raw data exists only in the user's object storage, and compaction/table maintenance are built in. But Tableflow agents do not expose the Kafka protocol. They are a separate table materialization cluster. Source: WarpStream Tableflow setup.

Key differences from Ursa:

  • WarpStream's main path is a diskless Kafka-compatible log, with lakehouse handled by Tableflow materialization.
  • Ursa's main path is a lakehouse-native storage engine, with topic/table unified at a lower level.
  • WarpStream's production adoption narrative is more mature. Ursa's stream-table duality is more aggressive.

If comparing "who is better for replacing a high-cost Kafka ingestion cluster", WarpStream is a first-tier competitor to Ursa. If comparing "who natively turns topics into Iceberg/Delta", WarpStream Tableflow looks more like Confluent Tableflow and is not a fully isomorphic competitor.

5.2 AutoMQ

AutoMQ describes itself as diskless Kafka on S3, 100% Kafka compatible, sub-10ms P99 latency, and zero cross-AZ costs. Sources: AutoMQ homepage, AutoMQ GitHub. Like Ursa, it targets the core cost problem of running Kafka in the cloud, but its technical trade-offs differ.

AutoMQ's performance benchmark docs clearly state that AutoMQ BYOC uses EBS as WAL storage to achieve low-latency send/receive. In other words, it is not simply an "all hot path writes directly to S3" model. It trades a WAL + shared/object storage hierarchy for latency. Source: AutoMQ Performance Benchmark. This explains why it can claim sub-10ms: the durable write path in the low-latency profile is not exactly the same comparison target as the object storage paths of WarpStream/Ursa.

AutoMQ is also adding lakehouse-native capabilities. The Table Topic docs say it can ingest streaming data into Iceberg tables, has a built-in schema registry, automatically creates Iceberg tables, supports AWS Glue / S3 Table Catalog, and reduces ETL pipelines such as Kafka Connect/Flink. Source: AutoMQ Table Topic.

Key differences from Ursa:

  • AutoMQ emphasizes Kafka compatibility, low latency, elasticity, and cost. Ursa emphasizes unified storage semantics between Kafka topics and lakehouse tables.
  • AutoMQ Table Topic appears close to Ursa's lakehouse goal, but whether it is strictly zero-copy and whether Kafka fetch and Iceberg query share the same Parquet objects must be verified by version and configuration.
  • AutoMQ's cross-AZ optimization has prerequisites, such as three AZs and balanced AZ deployment for producers/consumers and AutoMQ. The docs also say partial cross-AZ traffic appears when conditions are not met. Source: AutoMQ broker configuration.

AutoMQ is a competitor worth serious comparison with Ursa on both lines: Kafka-compatible diskless and lakehouse features.

5.3 Bufstream

Bufstream is one of Ursa's closest competitors on the lakehouse-native dimension. Public materials describe it as a Kafka-compatible broker / replacement that uses object storage and writes topic data as Iceberg archives in Parquet + Iceberg metadata. Parquet files simultaneously serve as data for Kafka consumers and as data files for Iceberg tables. The previously retrieved Bufstream docs explicitly call Iceberg Archives zero-copy and say no extra copy is created for Iceberg. Source: Bufstream Iceberg Archives reference.

Bufstream's distinctive point is schema/data quality: it relies on a Protobuf schema registry and places broker-side semantic validation plus Protobuf-to-Iceberg schema mapping in the write path. Source: Bufstream Iceberg configuration.

Key differences from Ursa:

  • Bufstream looks more like a "Kafka-compatible + Protobuf/schema-first + Iceberg storage" product path.
  • Ursa looks more like a "Kafka/Pulsar protocol + Lakestream storage layer + Oxia metadata + Iceberg/Delta" architecture path.
  • Both Bufstream and Ursa should be pressure-tested on consistency between Kafka semantics and Iceberg semantics, especially transactions, compacted topics, schema evolution, invalid records, and retention/snapshot lifecycle.

Because Bufstream pages redirected during the current retrieval, its product path and availability need separate confirmation. Architecturally, however, it is the most important lakehouse-native control group.

5.4 Aiven Iceberg Topics

Aiven Iceberg Topics is another important but easily overlooked path. It does not create a separate Kafka-compatible broker. Instead, it starts from Kafka tiered storage / the Remote Storage Manager plugin direction, directly persists topic segments as Parquet/Iceberg, and lets the same durable copy serve both Kafka replay and Iceberg analytics. The whitepaper recorded in the previous survey says that when a segment rotates, the RemoteStorageManager persists records as Parquet files into an Iceberg table, and fetch reconstructs valid Kafka batches. Source: Aiven Iceberg whitepaper.

Differences from Ursa:

  • Aiven is closer to Apache Kafka's internal extension points, with lower vendor/fork risk.
  • Ursa is a more complete external storage engine + metadata layer design, with stronger architectural control.
  • Aiven Iceberg Topics has the advantage of being Kafka-native. Its weakness is maturity and capability boundaries. The previous report noted that its README once marked Iceberg mode as recently added / alpha stage. Source: Aiven tiered-storage README.

If the comparison question is "can the Kafka upstream/plugin ecosystem naturally grow topic-as-table", Aiven Iceberg Topics is a key reference for Ursa.

5.5 Apache Kafka KIP-1150 / Aiven Inkless

KIP-1150 is Ursa's long-term standardization reference. Aiven's March 2026 article says KIP-1150 Diskless Topics has been accepted by the Apache Kafka community. It positions KIP-1150 as a meta/motivational KIP that first confirms whether Kafka should have diskless topics, with concrete implementation split into later sub-KIPs such as Diskless Core and Diskless Coordinator. Source: Aiven KIP-1150 accepted. Aiven's Inkless is a productized/experimental implementation of this direction. Aiven's blog describes the motivation as faster upgrades and recovery, lower infra/network costs, and says Inkless clusters are available in Aiven Cloud. Sources: Aiven Inkless announcement, Aiven Diskless Kafka.

KIP-1150's direction is similar to Ursa: a diskless topic does not use broker disk as the primary durable storage for user data, Kafka delegates replication to object storage, and brokers are no longer the only partition leaders. KIP-1163 further defines the produce/fetch path, KIP-1164 covers the diskless coordinator, and KIP-1183 proposes a unified shared storage / shared log abstraction. Sources: KIP-1150, KIP-1163, KIP-1164, KIP-1183.

Key differences from Ursa:

  • KIP-1150 is an Apache Kafka upstream direction, not a complete product.
  • It mainly solves diskless topics and does not build in Ursa-style Iceberg/Delta table duality.
  • If KIP-1150 matures, some of the differentiation of Ursa's Kafka fork will be compressed. But Ursa's lakehouse-native/cross-protocol Lakestream may still remain differentiated.
  • Aiven's public benchmarks provide a practical latency/cost profile for diskless topics. These numbers can serve as references for an Ursa POC, but should not be mixed directly with Ursa's lakehouse-native table path. Source: Aiven benchmarking diskless topics.

If your core need is "follow Kafka upstream and reduce long-term fork/vendor risk", KIP-1150/Aiven Inkless is an important reference. If your core need is "turn topics into lakehouse tables now", it is not yet a direct substitute.

5.6 Confluent Freight + Tableflow

Confluent represents the conservative enterprise path. Freight clusters direct write data to object storage, avoiding local storage and replication in Kora brokers. Confluent positions it as a low-cost cluster type where latency can be relaxed, up to a second or two. Source: Confluent Freight GA.

Tableflow materializes Kafka topics into Iceberg/Delta tables. Confluent docs define it as Kafka topics as Iceberg and Delta Lake tables. The GA blog says it converts Kafka messages into Parquet, organizes them into Iceberg tables, and manages metadata/maintenance such as compaction. Sources: Confluent Tableflow docs, Tableflow GA blog.

Key differences from Ursa:

  • Confluent's path splits "Kafka storage" and "Kafka-to-table materialization" into two product surfaces: Freight + Tableflow.
  • Ursa combines both into one lakehouse-native storage engine.
  • Confluent has strong maturity, SLA, ecosystem, and governance tooling. Ursa's architecture is cleaner but newer and more aggressive.

In real procurement, Confluent may be the conservative substitute Ursa encounters most often: not isomorphic, but the lowest acceptable-risk path for enterprise users.

5.7 Redpanda Tiered Storage / Cloud Topics

Redpanda is not an Ursa-style lakehouse-native system, but it is an important reference for Kafka API-compatible alternatives and the object-storage trend. Redpanda Tiered Storage docs say it offloads log segments to object storage to reduce storage cost and accesses remote data through remote write / remote read. Source: Redpanda Tiered Storage.

Redpanda's 2026 Cloud Topics engineering posts focus on two practical issues for streaming on object storage: how high-latency storage maintains throughput through concurrency under Little's Law, and lifecycle/garbage collection for Level Zero temporary objects. Sources: Little's Law in practice with Cloud Topics, Cloud Topics: Level Zero garbage collection.

Key differences from Ursa:

  • Redpanda prioritizes high-performance Kafka-compatible streaming, not topic-as-Iceberg-table.
  • It is suitable for comparing low latency, operations, Kafka compatibility, and tiered/object storage maturity with Ursa.
  • If the workload is still operational streaming rather than lakehouse ingestion, Redpanda may be a more natural fit than Ursa.

5.8 Fluss

Fluss provides another counter-reference: it does not argue that Iceberg should directly carry all hot streaming storage. Instead, it treats Iceberg/Paimon as the lakehouse/cold tier and keeps low-latency streaming, primary-key semantics, and real-time updates in dedicated hot streaming storage. The previous survey recorded its judgment: Iceberg is better suited as a cold tier, while Fluss provides hot streaming storage, sub-second latency, primary-key semantics, and then tiers into Iceberg/Paimon. Source: Fluss x Iceberg.

It is not a direct procurement competitor to Ursa, but it matters for technical judgment. Ursa's implicit judgment is that streaming logs and lakehouse tables can be unified deeply enough. Fluss's judgment is that hot streams and lakehouse tables should be layered and connected at the boundary. If the workload has high-frequency updates, low-latency primary-key reads/updates, or complex changelogs, a layered path such as Fluss may be more natural than Ursa-style topic-as-table.

6. Workloads Where Ursa Fits and Does Not Fit

6.1 Ursa Fits

  • High-throughput logs, metrics, clickstream, IoT ingestion.
  • CDC / data lake ingestion.
  • Raw event landing for AI/ML pipelines.
  • Scenarios that need longer retention and want to reduce Kafka local disk and connector pipelines.
  • Scenarios that can tolerate 200ms-to-seconds freshness rather than sub-50ms tail latency.
  • Scenarios where Spark/Trino/Flink/Snowflake/Databricks should read streaming data directly.

6.2 Ursa Does Not Fit

  • Low-latency transactions, online risk control, request-response event buses.
  • Complex workloads that heavily depend on Kafka compaction / tombstones / transactions / EOS, unless the POC covers them well.
  • Small-message, low-concurrency workloads with strict tail-latency SLOs.
  • Teams unwilling to take on a new metadata/catalog control plane.
  • Teams that already have a mature Kafka + Connect/Flink + Iceberg pipeline and whose cost is not a primary bottleneck.

7. How to Test Ursa in a POC

7.1 Performance and Cost

  • Use real message size, partition count, producer concurrency, and consumer fanout rather than only the standard OpenMessaging happy path.
  • Measure p50/p95/p99/p999 produce latency, E2E latency, and catch-up read latency separately.
  • Test both low-concurrency small batches and high-concurrency large batches, because object storage systems are highly sensitive to Little's Law.
  • Cost must be separated into broker, Oxia/metadata, compaction workers, catalog, object storage PUT/GET/list/range requests, S3 Express, cross-AZ/cross-region, and egress.

7.2 Kafka Semantics

  • transactional producer / exactly-once.
  • consumer group churn, rebalance, offset commit.
  • compacted topics, tombstones, retention delete.
  • key skew, hot partitions, multi-tenant isolation.
  • broker crash, metadata leader failover, object storage throttling, AZ failure.

7.3 Lakehouse Semantics

  • Iceberg/Delta table freshness SLA.
  • small-file count, compaction lag, snapshot count, manifest growth.
  • schema evolution, high-frequency schema changes, invalid records / DLQ.
  • read consistency for external query engines: whether Spark/Trino/Snowflake/Databricks see the expected snapshot.
  • catalog lock contention and catalog vendor lock-in.

8. Decision Framework

If the goal is lowering Kafka cost while still mainly serving Kafka consumers, prioritize comparing WarpStream, AutoMQ, Confluent Freight, Aiven/KIP-1150, and Redpanda Cloud Topics. Ursa can be included, but should not receive credit only because it is lakehouse-native.

If the goal is turning Kafka ingestion directly into Iceberg/Delta data assets, prioritize comparing Ursa, Bufstream, AutoMQ Table Topic, WarpStream Tableflow, Confluent Tableflow, and Aiven Iceberg Topics. Here Ursa is one of the most architecturally ambitious options.

If the goal is safe enterprise deployment, Confluent Cloud/MSK/Kafka + Tableflow/Connect/Flink remains the realistic baseline. It may be less elegant and more expensive, but maturity, SLA, ecosystem, and hiring/operations familiarity are stronger.

If the goal is judging the long-term technical direction, KIP-1150/1163/1164 matter more than any single vendor. Once upstream Kafka forms a diskless topic standard, vendor differentiation will move from "can it be diskless" to "how good are metadata, catalog, maintenance, governance, and lakehouse integration".

9. Ursa Bear Case: Why It May Not Be Right for You

If you only read StreamNative's paper and launch posts, Ursa can easily look like a unified answer to Kafka cost, broker operations, lakehouse ingestion, and stream-table duality. From the bear-case view, it has at least seven ways to go wrong.

9.1 It May Only Move Kafka's Complexity Somewhere Else

Ursa removes the partition leader, ISR, local log, and rebalancing from Kafka brokers, but it does not remove hard problems such as ordering, offset assignment, transaction visibility, fencing, and metadata commits. They move into Oxia, the offset index, catalog commits, and the compaction service.

This means Ursa's operational problem is no longer "which broker disk is full, which partition leader is hot". It becomes:

  • whether Oxia/metadata shards become hot partitions.
  • whether metadata quorum cross-AZ cost and latency are fully accounted for.
  • whether catalog commits become a new bottleneck under high-frequency writes.
  • whether the offset index and table metadata remain consistent during failure recovery.
  • whether Kafka replay and table queries degrade together when compaction falls behind.

So Ursa's complexity does not disappear. It moves from the familiar Kafka broker layer into a newer metadata/table-control-plane layer. For a team, this may reduce dimensionality, or it may replace familiar failure modes with a more unfamiliar set.

9.2 Its Best-Fit Scenario Is Narrower Than the Marketing Narrative

Ursa is best suited for high-throughput data streams that can tolerate sub-second freshness and are centered on lake landing / analytics / AI ingestion. If the workload is mainly operational streaming, such as low-latency event buses, order status streams, risk-control signals, online service coordination, or strict p99/p999 SLOs, Ursa's lakehouse-native advantage may go unused while the cost of object storage and metadata commits becomes exposed.

Put differently, Ursa's question is not "can it run the Kafka API", but "does your Kafka workload really want to trade 200-500ms or higher tail latency for cost and lakehouse visibility". If the answer is no, Redpanda, traditional Kafka, Confluent low-latency profiles, AutoMQ low-latency profiles, or WarpStream S3 Express profiles may be more reasonable.

9.3 Topic-as-Table May Underestimate Iceberg/Delta Maintenance Cost

"Every Kafka topic is an Iceberg/Delta table" sounds clean, but production Iceberg/Delta tables are not just Parquet files. They also require:

  • small-file compaction and compaction backlog handling.
  • snapshot expiration and orphan file cleanup.
  • schema evolution and bad-data handling.
  • partition/sort strategy effects on query performance.
  • catalog commit contention.
  • semantic boundaries between table maintenance and Kafka retention.

Traditional Kafka-to-Iceberg pipelines are ugly, but at least their complexity is explicit: Connect/Flink/Spark jobs, DLQs, catalogs, compaction, and monitoring are each visible. Once Ursa internalizes this pipeline, the operational surface shrinks, but the system internals must get all of these things right. For users, this may reduce operational burden, or it may reduce observability and control.

9.4 The Hard Part of Kafka Compatibility Is Long-Tail Semantics, Not the Happy Path

Produce/fetch, consumer groups, and ordinary append-only topics do not prove that Ursa is equivalent to Kafka. The semantics that really need pressure-testing are long-tail cases:

  • transactions / exactly-once.
  • compacted topic / tombstone.
  • key skew and hot partitions.
  • high fan-out catch-up readers.
  • interactions between retention delete and the Iceberg snapshot lifecycle.
  • consumer group churn and offset commit storms.
  • schema evolution storms.
  • recovery under object storage throttling, catalog outage, and metadata failover.

These are also the scenarios vendor benchmarks are most likely to under-cover. If a POC only measures throughput and average latency, Ursa's risk will be systematically underestimated.

9.5 Cost Advantages May Be Partly Consumed by the Control Plane and Table Maintenance

Ursa's cost savings mainly come from reducing cross-AZ broker replication, local disk, and connector pipelines. But the real bill also needs to include:

  • Oxia/metadata service machines, storage, and quorum traffic.
  • compaction workers.
  • object storage PUT/GET/range request/list cost.
  • Iceberg/Delta catalog and table maintenance.
  • amplification when query engines read unoptimized small files.
  • the cost of using a more expensive storage tier or smaller batches to lower latency.

This does not mean Ursa cannot save money. It means conclusions such as "10x cheaper" must be itemized on your own workload. Especially when there are many topics, small messages, frequent schema changes, or high query freshness requirements, metadata and maintenance costs will stand out more than they do in marketing materials.

9.6 The Fork Path Carries Long-Term Maintenance Risk

One of UFK's selling points is that it is a Kafka fork, not a proxy. But the fork is also a risk:

  • how UFK follows upstream Kafka as KIP-1150/1163/1164 continue to evolve.
  • how Kafka security patches, protocol changes, and group coordinator/KRaft behavior changes are synchronized.
  • whether surrounding tools that depend on broker internal behavior, rather than pure protocol behavior, see compatibility differences.
  • whether UFK's differentiation shifts toward a proprietary storage/catalog layer if native Apache Kafka diskless topics mature in the future.

These risks may not affect a short-term POC, but they matter for a 3-5 year platform choice.

9.7 The Biggest Uncertainty Is Maturity, Not Whether the Architecture Is Elegant

Ursa's architecture direction is persuasive and backed by a PVLDB paper. But for platform teams, the question is not "is this architecture advanced". It is:

  • whether there are enough public production cases.
  • whether Oxia, the offset index, compaction, and catalog commits can be observed and debugged.
  • whether there are clear failure recovery runbooks.
  • whether the Kafka semantics and governance tools you depend on are supported.
  • whether it can run stably in your cloud, your object storage, your catalog, and your query engine combination.

Therefore, Ursa is better suited as a high-potential POC candidate than as an immediate default Kafka platform replacement. The safest adoption path is to start with high-throughput, low-latency-insensitive, append-heavy topics where lake landing has high value, rather than migrating core operational topics.

10. Final Conclusion

Ursa is worth serious study because it compresses three trends into one architecture:

  • Kafka brokers becoming stateless / diskless.
  • Cloud object storage as the durable streaming log.
  • Iceberg/Delta catalogs as the table control plane for streaming data.

Its benefits are clear: it saves cross-AZ replication and local disk, simplifies broker operations, and turns the Kafka-to-lakehouse pipeline into a native storage path. Its risks are equally clear: the metadata/control plane becomes a new hot path, object storage latency and request cost enter the system design, Iceberg/Delta maintenance is no longer a side-path problem, and the Kafka fork/product maturity must be verified.

The most accurate assessment is: Ursa is a strong template for a lakehouse-native streaming engine, not a universal replacement for every Kafka workload. For high-throughput, cost-sensitive analytics/AI/CDC lake landing scenarios, it is a very attractive direction. For low-latency operational streaming, traditional Kafka/Redpanda or low-latency profiles from AutoMQ/Confluent/WarpStream may be more suitable.

More sharply: Ursa's greatest value is that it turns Kafka into part of the lakehouse ingestion/control plane. Its greatest risk is the same thing. You are not just getting "cheap Kafka". You are adopting a new streaming-table storage engine. Ursa is the better choice only when the benefits of lakehouse-native design are large enough to cover metadata, catalog, compaction, fork, and maturity risks.

If continuing this work, the most valuable next step would be a POC checklist or benchmark matrix that puts Ursa, WarpStream, AutoMQ, and Bufstream/Confluent Tableflow under the same workload, rather than further comparing vendor claims.
