Fivetran 与 dbt Labs 合并：数据栈整合的机会，以及 dbt Core 的真正风险

Fivetran 和 dbt Labs 的交易不应该被简单理解成“Fivetran 收购 dbt”。更准确地说，这是两家现代数据栈核心公司在公开市场窗口不确定、AI 叙事重塑软件估值、数据平台竞争加剧的背景下，用全股票方式合并成一个更大的数据基础设施平台。

这笔交易的短期重点是规模和产品拼图：数据移动、转换、元数据、语义上下文、激活和质量控制被放进同一个 “Open Data Infrastructure for AI agents” 叙事。长期真正要看的，是它会不会把 dbt 的开放生态升级成更强的开放标准，还是把开放叙事变成一个更大的商业控制面。

交易事实

Fivetran 在 2025 年 10 月 13 日宣布与 dbt Labs 签署 definitive agreement，以全股票方式合并。官方公告称，合并后 George Fraser 将担任 CEO，dbt Labs CEO Tristan Handy 将担任 co-founder and President，合并公司 ARR 接近 6 亿美元。来源：Fivetran 官方公告。

Reuters 的报道补充了交易结构：换股比例与收入和增长率相关；Fivetran 最近一次公开估值是 2021 年 56 亿美元，dbt Labs 最近一轮是 2022 年 42 亿美元；双方共同投资人包括 Andreessen Horowitz。来源：Reuters via Yahoo Finance。

“The deal is structured as an all-stock exchange based on an agreed ratio tied to revenues and growth rates...”

2026 年 6 月 1 日，Fivetran 宣布合并完成。合并后初期以 Fivetran + dbt Labs 运营，并称服务全球超过 100,000 个数据团队。来源：Fivetran 完成合并公告。

为什么这笔交易发生

最直接的商业逻辑是客户和工作流高度重叠。Fivetran 管数据进入仓库或湖仓，dbt 管数据转换、测试、文档和语义建模。Reuters 引用 George Fraser 的说法，估计 Fivetran 客户中 80% 到 90% 使用 dbt 工具。来源：Reuters via Yahoo Finance。

“Fraser estimated that 80% to 90% of Fivetran's customers use dbt's tools.”

如果这个比例接近真实，合并就不只是横向规模扩张，而是把企业数据管道中最常一起出现的两个层合并成一个销售和产品平台。

更深一层，是现代数据栈正在从 “best-of-breed 工具拼装” 转向更少 vendor、更强集成和更清晰责任边界。过去几年，企业数据团队往往用 Fivetran/Airbyte 做 ingestion，用 dbt 做 transformation，用 Airflow/Dagster 做 orchestration，用 Monte Carlo/Great Expectations 做 quality，用 Census/Hightouch 做 activation。这个组合灵活，但也带来合同、权限、lineage、incident ownership 和成本归因的复杂度。

Fivetran 的新叙事是：企业 AI agents 需要一个可靠、新鲜、治理过、带语义上下文的数据基础层。合并完成公告中，Fivetran 明确说 AI agents 正在成为企业数据的主要消费者，它们连续、并行、以机器速度运行，因此底层数据必须 reliable、fresh、governed、accessible。来源：Fivetran 完成合并公告。

“AI agents are quickly becoming the primary consumers of enterprise data... requiring it to be reliable, fresh, governed, and accessible across every system in the enterprise.”

这让 Fivetran + dbt 的故事从“帮你做 analytics pipeline”升级成“帮 AI agents 可信地使用企业数据”。

Fivetran 在补完整个数据生命周期

这笔合并不是孤立事件。Fivetran 在 2025-2026 年连续扩张到数据栈其他层：Census 代表 activation / reverse ETL，Tobiko 代表 SQLMesh / SQLGlot 和高级 transformation，Great Expectations / GX Core steward 计划代表 data quality，dbt Labs 则代表 transformation 标准和 analytics engineering 社区。

Gradient/MarTech 把这称为 Fivetran 五个月内第三笔并购或合并，并指出此前包括 Census 和 Tobiko Data。来源：Gradient Group / MarTech。

从战略上看，Fivetran 不再满足于只做 ELT 里的 E 和 L。它想控制从数据进入、变换、校验、解释、激活到 AI context 的完整控制面。

这也解释了为什么它反复强调 “open”。如果 Fivetran 只是做一个封闭套件，它会直接撞上 Snowflake、Databricks、Google、AWS 和 Microsoft 的平台控制力。它必须把自己包装成跨云、跨引擎、跨 catalog、跨 BI 和 AI 工具的中立基础设施层。Fivetran 官方博客强调 open table formats、shared metadata、decoupling storage from compute，以及 across clouds, engines, catalogs 的互操作。来源：Fivetran 产品博客。

“Open standards have matured... decoupling storage from compute and enabling reliable interoperability across clouds, engines, and catalogs.”

dbt Core 的风险不是明天改许可证

交易宣布时，Fivetran 承诺保持 dbt Core 在当前许可证下开放，并与社区一起维护。来源：Fivetran 官方公告。

“the company is committed to keeping dbt Core open under its current license and maintaining it with and for the community”

Fivetran 同日产品博客说得更明确：dbt Core 将继续按当前许可证开源并无限期支持维护；dbt Fusion engine 也会继续 source-available；SQLMesh 和 SQLGlot 继续开源。来源：Fivetran 产品博客。

“dbt Core will remain open source under its current license and will continue to be supported and maintained indefinitely.”

更重要的是，合并完成当天 dbt 发布 dbt Core v2.0 alpha，称已经把大量 Fusion runtime 代码放进 dbt-core，并以 Apache 2.0 开源。来源：dbt Developer Blog。

“Today, we published the first alpha release of dbt Core version 2.0... dbt Core's code is completely open source under the Apache 2.0 license.”

这是一个很强的正面信号。它说明 Fivetran + dbt 没有只做 “旧 Core 维持不死” 的低配承诺，而是把新的 Rust/Fusion runtime 核心部分给了开源 Core。

但这不等于风险消失。真正的风险从 “dbt Core 明天会不会改许可证” 变成了三个更慢、更实际的问题：

  1. Core 是否会继续得到实质创新，还是只保留为合规和社区信任的基础层？
  2. 最好的开发体验、agent workflow、metadata/context 和 cost optimization 是否逐渐集中到 Fusion / dbt platform / Fivetran bundle？
  3. dbt Core 的治理是否仍是 company-controlled OSS，还是会走向更中立、更可参与的治理？

dbt 自己的 v2.0 文章也承认，官方认为大多数用户应该安装 Fusion，因为它是更好的免费本地和生产 CLI，并且有更多能力；部分高级功能需要免费登录或付费账号解锁。来源：dbt Developer Blog。

“For the best free dbt CLI you can use locally and in production environments, install Fusion... some are totally free to use, and other premium features are unlocked with a free login or payment method.”

所以要看的不是许可证口号，而是未来两年 Core、Fusion、Cloud 和 Fivetran 平台之间的功能分配。

外部批评的合理部分

Nexla 的批评是，Apache 2.0 只能保护已经发布的代码，不能保证未来创新、品牌、社区空间和 roadmap。来源：Nexla。

“The license protects what's already released. It doesn't guarantee future innovation.”

这句话是对的。开源许可证不是产品路线图，也不是治理机制。Fivetran 现在控制的不只是代码，还有 dbt 品牌、主仓库、文档、社区入口、商业销售路径和产品默认分发。

Datacoves 的观点更克制：它相信 dbt Core 许可证不会改变，因为改许可证会造成社区分裂和 goodwill 损失；但它也认为未来重心不是 dbt Core，而是 Fusion。来源：Datacoves。

“dbt Labs and Fivetran have both stated that dbt Core's license would not change, and I believe them... The future vision for dbt is not dbt Core, but instead dbt Fusion.”

我更接近这个判断：短期 license panic 可能过度了，但长期商业重心偏移是真问题。

第一批联合产品透露了方向

合并完成时，Fivetran + dbt Labs 同时发布了几类新东西：

  1. dbt Core v2.0 alpha：Fusion runtime 进入 Apache 2.0 Core。
  2. dbt State：只 build 发生变化的部分，官方称可降低底层基础设施成本 30% 或更多。
  3. dbt Wizard：面向 dbt 项目的 agent，做模型编写、重构和调试。
  4. Agents Schema：把 metric definitions、semantic models、dbt lineage 和业务文档放进 SQL tables，作为 AI agents 的共享上下文层。

来源：Fivetran 完成合并公告、Fivetran 产品博客。

这些产品说明，Fivetran + dbt 不是只想把 EL 和 T 接起来。它想把 “数据如何被 agent 理解、验证、跳过重复计算、生成 SQL、追踪 lineage” 做成新的控制面。

这就是机会，也是风险。机会是数据团队真的需要更少 glue code 和更好的上下文层。风险是上下文层一旦变成 vendor-controlled，迁移成本会比 connector 或 SQL model 高得多。

谁受益，谁承压

短期受益最明显的是已经同时使用 Fivetran 和 dbt 的企业。它们可能得到更顺的合同、更紧的产品集成、更少的责任扯皮，以及一个更清晰的 AI data foundation 采购故事。

Fivetran 和 dbt 自己也受益。合并后的 ARR 规模更接近未来 IPO 候选体；Fivetran 获得 dbt 的品牌、社区和 transformation 标准；dbt 获得更强的企业销售、数据移动能力和平台资源。

承压的是独立工具生态。Airbyte、dlt、Hightouch、Dagster、Kestra、Dataform、SQLMesh、各类 data quality 和 semantic layer 工具，都要面对一个更完整、更有销售资源的 Fivetran + dbt。与此同时，它们也会拿 “保持开放、避免锁定” 作为反击点。

Snowflake、Databricks 和云厂商的关系更复杂。Fivetran + dbt 既是它们的生态伙伴，也是潜在控制面竞争者。如果 Fivetran + dbt 成为跨平台 business logic 和 agent context 层，warehouse/lakehouse 的差异化就会被部分抽象掉；但如果云厂商把更多 ingestion、transformation、governance 和 AI agent 功能内建进去，Fivetran + dbt 也会被挤压。

给数据团队的实际建议

短期不用恐慌迁移。dbt Core 没有消失，v2.0 alpha 反而是积极信号。现有 Fivetran + dbt 工作流也不会因为合并立刻变坏。

但采购和架构上要更清醒：

  1. 观察 Core release 节奏、社区 PR 响应、Fusion / Cloud 独占功能比例、bundle pricing，而不是只看 “open” 的口号。
  2. 把关键业务逻辑保留在 Git、标准 SQL、开放表格式和可替换 orchestration 里。
  3. 如果购买 Fivetran + dbt bundle，要问清楚功能边界、价格保护、metadata 导出、合同结束后的运行权利。
  4. 不要把 agent context、semantic layer、lineage 和 quality signals 全部锁进单一 SaaS 控制面，除非商业收益足够明确。
  5. 关注 dbt Core 是否出现更中立的 governance。如果长期仍完全 company-controlled，fork 权利虽然存在，但实际社区迁移成本会很高。

最终判断

这笔合并是现代数据栈的一次重要拐点。Fivetran 买到的不只是 dbt 的技术，而是 transformation 标准、社区、品牌和客户入口；dbt 得到的也不只是一个新母体，而是更强的企业分发和 AI data infrastructure 叙事。

最可能的 base case 是：Fivetran + dbt 会继续维持 dbt Core 的开源合法性和基本活跃度，同时把最强商业化能力放到 Fusion、dbt platform、Fivetran managed services 和 AI/agent 产品上。

所以真正值得盯的不是 “dbt Core 明天会不会改许可证”。更重要的问题是：两年后，最好的 dbt 体验、最完整的 metadata/context、最低的运行成本和最强的 agent workflow，是否仍然能在开放、可替换、可自托管的架构中实现？

如果答案逐渐变成 “必须买 Fivetran + dbt 的商业栈”，这笔交易就是开放叙事下的平台锁定。如果 dbt Core v2、Agents Schema、Iceberg/SQL 标准和第三方 managed Fusion 真的繁荣，它才可能成为数据基础设施开放标准的一次整合升级。

Fivetran and dbt Labs Merge: The Opportunity in Data Stack Consolidation, and the Real Risk for dbt Core

The Fivetran and dbt Labs transaction should not be reduced to “Fivetran bought dbt.” A more accurate reading is that two core modern data stack companies merged through an all-stock transaction at a moment when the IPO window remains uncertain, AI is reshaping software valuations, and data platform competition is intensifying.

The near-term story is scale and product completeness: data movement, transformation, metadata, semantic context, activation, and quality are being pulled into one “Open Data Infrastructure for AI agents” narrative. The long-term question is whether this turns dbt's open ecosystem into a stronger open standard, or whether the open narrative becomes a larger commercial control plane.

The Transaction

On October 13, 2025, Fivetran announced that it had signed a definitive agreement to merge with dbt Labs in an all-stock deal. The company said George Fraser would serve as CEO, dbt Labs CEO Tristan Handy would become co-founder and President, and the combined company would approach $600 million in ARR. Source: Fivetran announcement.

Reuters added more detail on the transaction structure: the stock exchange ratio was tied to revenue and growth rates; Fivetran was last publicly valued at $5.6 billion in 2021, while dbt Labs was valued at $4.2 billion in 2022; and the companies share investors including Andreessen Horowitz. Source: Reuters via Yahoo Finance.

“The deal is structured as an all-stock exchange based on an agreed ratio tied to revenues and growth rates...”

On June 1, 2026, Fivetran announced that the merger had closed. The combined company will initially operate as Fivetran + dbt Labs and says it serves more than 100,000 data teams globally. Source: Fivetran merger completion announcement.

Why the Deal Happened

The most direct logic is workflow overlap. Fivetran moves data into warehouses and lakehouses; dbt transforms, tests, documents, and models that data. Reuters quoted George Fraser saying he estimated that 80% to 90% of Fivetran customers use dbt tools. Source: Reuters via Yahoo Finance.

“Fraser estimated that 80% to 90% of Fivetran's customers use dbt's tools.”

If that estimate is close, the merger is not only about horizontal scale. It combines two layers that already show up together in enterprise data pipelines into one product and sales platform.

The deeper reason is that the modern data stack is moving away from a pure best-of-breed assembly model toward fewer vendors, tighter integration, and clearer responsibility boundaries. Over the past few years, data teams commonly used Fivetran or Airbyte for ingestion, dbt for transformation, Airflow or Dagster for orchestration, Monte Carlo or Great Expectations for quality, and Census or Hightouch for activation. That stack is flexible, but it also creates complexity around contracts, permissions, lineage, incident ownership, and cost attribution.

Fivetran's new narrative is that enterprise AI agents need a reliable, fresh, governed, semantically rich data foundation. In the merger completion announcement, Fivetran said AI agents are becoming primary consumers of enterprise data and operate continuously, in parallel, and at machine speed; therefore the underlying data must be reliable, fresh, governed, and accessible. Source: Fivetran merger completion announcement.

“AI agents are quickly becoming the primary consumers of enterprise data... requiring it to be reliable, fresh, governed, and accessible across every system in the enterprise.”

That reframes Fivetran + dbt from an analytics pipeline vendor into a trusted data layer for enterprise AI agents.

Fivetran Is Filling Out the Data Lifecycle

The dbt Labs merger is not an isolated move. In 2025 and 2026, Fivetran expanded into multiple other parts of the data stack: Census for activation and reverse ETL, Tobiko for SQLMesh / SQLGlot and advanced transformation, Great Expectations / GX Core stewardship for data quality, and dbt Labs for the transformation standard and analytics engineering community.

Gradient / MarTech described the dbt transaction as Fivetran's third merger or acquisition in five months, after Census and Tobiko Data. Source: Gradient Group / MarTech.

Strategically, Fivetran is no longer trying to own only the E and L in ELT. It wants to control the broader plane from ingestion to transformation, validation, explanation, activation, and AI context.

That also explains why it keeps emphasizing “open.” If Fivetran were simply building a closed suite, it would run directly into Snowflake, Databricks, Google, AWS, and Microsoft. It has to position itself as a cross-cloud, cross-engine, cross-catalog, cross-BI and AI-neutral infrastructure layer. Fivetran's product blog emphasized open table formats, shared metadata, decoupling storage from compute, and interoperability across clouds, engines, and catalogs. Source: Fivetran product blog.

“Open standards have matured... decoupling storage from compute and enabling reliable interoperability across clouds, engines, and catalogs.”

The Real dbt Core Risk Is Not a License Change Tomorrow

When the deal was announced, Fivetran committed to keeping dbt Core open under its current license and maintaining it with the community. Source: Fivetran announcement.

“the company is committed to keeping dbt Core open under its current license and maintaining it with and for the community”

Fivetran's same-day product blog was even more explicit: dbt Core would remain open source under its current license and continue to be supported and maintained indefinitely; the dbt Fusion engine would remain source-available; SQLMesh and SQLGlot would remain open source. Source: Fivetran product blog.

“dbt Core will remain open source under its current license and will continue to be supported and maintained indefinitely.”

More importantly, on the day the merger closed, dbt announced dbt Core v2.0 alpha, saying it had moved a large amount of Fusion runtime code into dbt-core under Apache 2.0. Source: dbt Developer Blog.

“Today, we published the first alpha release of dbt Core version 2.0... dbt Core's code is completely open source under the Apache 2.0 license.”

That is a strong positive signal. It shows Fivetran + dbt did not merely promise to keep the old Core alive. They moved important new Rust / Fusion runtime code into the open source Core.

But that does not remove the risk. It changes the risk from “will dbt Core's license change tomorrow?” to three slower, more practical questions:

  1. Will Core keep receiving real innovation, or will it become the compliance and community-trust baseline?
  2. Will the best developer experience, agent workflows, metadata/context, and cost optimization move toward Fusion, dbt platform, and Fivetran bundles?
  3. Will dbt Core remain company-controlled open source, or move toward more neutral, participatory governance?

dbt's own v2.0 post also says the company expects most users to install Fusion because it is the best free local and production CLI, includes more capabilities, and can unlock some advanced features through free login or paid accounts. Source: dbt Developer Blog.

“For the best free dbt CLI you can use locally and in production environments, install Fusion... some are totally free to use, and other premium features are unlocked with a free login or payment method.”

So the issue to watch is not the license slogan. It is how features are allocated across Core, Fusion, Cloud, and the Fivetran platform over the next two years.

The Valid Part of the External Criticism

Nexla's criticism is that Apache 2.0 protects already-released code, but does not guarantee future innovation, the brand, community spaces, or the roadmap. Source: Nexla.

“The license protects what's already released. It doesn't guarantee future innovation.”

That point is correct. An open source license is not a product roadmap or a governance model. Fivetran now controls not only the code, but also the dbt brand, main repositories, documentation, community entry points, commercial sales motion, and default distribution path.

Datacoves takes a more measured view. It says it believes dbt Core's license will not change because doing so would fragment the community and destroy goodwill, but also argues that the future vision for dbt is not Core, but Fusion. Source: Datacoves.

“dbt Labs and Fivetran have both stated that dbt Core's license would not change, and I believe them... The future vision for dbt is not dbt Core, but instead dbt Fusion.”

That is close to my view. Short-term license panic is probably overdone. Long-term commercial center-of-gravity drift is real.

The First Joint Products Show the Direction

When the merger closed, Fivetran + dbt Labs announced several products:

  1. dbt Core v2.0 alpha: Fusion runtime enters Apache 2.0 Core.
  2. dbt State: builds only what changed; Fivetran says it can reduce underlying infrastructure cost by 30% or more.
  3. dbt Wizard: an agent for dbt projects that helps with model authoring, refactoring, and debugging.
  4. Agents Schema: a SQL-table-based context layer for metric definitions, semantic models, dbt lineage, and business documentation.

Sources: Fivetran merger completion announcement, Fivetran product blog.

These products show that Fivetran + dbt is not merely connecting EL and T. It is trying to define a new control plane for how agents understand, validate, skip redundant compute, generate SQL, and trace lineage.

That is both the opportunity and the risk. Data teams do need less glue code and a better context layer. But if that context layer becomes vendor-controlled, the switching cost may be much higher than the switching cost of a connector or a SQL model.

Who Benefits, Who Comes Under Pressure

The most obvious near-term beneficiaries are enterprises already using both Fivetran and dbt. They may get cleaner contracting, tighter product integration, less incident finger-pointing, and a clearer AI data foundation procurement story.

Fivetran and dbt also benefit. The combined ARR base looks more like a future IPO candidate; Fivetran gets dbt's brand, community, and transformation standard; dbt gets stronger enterprise distribution, data movement capabilities, and a larger platform narrative.

The pressure falls on the independent tool ecosystem. Airbyte, dlt, Hightouch, Dagster, Kestra, Dataform, SQLMesh, data quality vendors, and semantic layer vendors now face a more complete and better-resourced Fivetran + dbt. At the same time, they will use “stay open and avoid lock-in” as their counter-positioning.

The relationship with Snowflake, Databricks, and cloud vendors is more complex. Fivetran + dbt is both ecosystem partner and potential control-plane competitor. If it becomes the cross-platform layer for business logic and agent context, some warehouse and lakehouse differentiation gets abstracted away. But if cloud data platforms keep building ingestion, transformation, governance, and AI-agent features natively, Fivetran + dbt will face pressure from below and above.

Practical Advice for Data Teams

There is no need to panic-migrate in the short term. dbt Core is not disappearing, and v2.0 alpha is a positive signal. Existing Fivetran + dbt workflows will not break just because the merger closed.

But procurement and architecture decisions should become more deliberate:

  1. Watch Core release cadence, community PR response, the ratio of Fusion / Cloud-exclusive features, and bundle pricing, not just “open” messaging.
  2. Keep critical business logic in Git, standard SQL, open table formats, and replaceable orchestration.
  3. If buying a Fivetran + dbt bundle, ask about feature boundaries, price protection, metadata export, and runtime rights after contract expiration.
  4. Do not lock all agent context, semantic layer, lineage, and quality signals into a single SaaS control plane unless the business value is clearly worth it.
  5. Watch whether dbt Core develops more neutral governance. If it remains fully company-controlled, fork rights exist in theory, but the practical community migration cost will be high.

Final View

This merger is an important turning point for the modern data stack. Fivetran did not only get dbt's technology. It got the transformation standard, the community, the brand, and the customer entry point. dbt did not only get a new parent company. It got stronger enterprise distribution and a larger AI data infrastructure narrative.

The most likely base case is that Fivetran + dbt will keep dbt Core open and reasonably active, while putting the strongest commercial energy into Fusion, dbt platform, Fivetran managed services, and AI/agent products.

So the question to watch is not “will dbt Core's license change tomorrow?” The better question is: two years from now, can the best dbt experience, the richest metadata/context layer, the lowest operating cost, and the strongest agent workflow still be achieved in an open, replaceable, self-hostable architecture?

If the answer gradually becomes “you need to buy the Fivetran + dbt commercial stack,” then this deal becomes platform lock-in under an open narrative. If dbt Core v2, Agents Schema, Iceberg / SQL standards, and third-party managed Fusion all thrive, then it could become a genuine consolidation upgrade for open data infrastructure.
